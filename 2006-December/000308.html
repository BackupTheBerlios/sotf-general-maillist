hey matthias + othe video node coders,

some weeks ago i did some testing of the video node code with my own
install at http://video.xaok.org

well basically i did some tuning on ffmpeg parameters to render
h.264 / avc comliant output and better quality divx output
(if that is of interest for the list i can post my config changes)
and updated to getid3-1.7.7

after christmas until end of january i could do some more serious
testing and bug reporting on the video part. is somebody willing
to do some coding/bugfixing on the backend side?

in short, most problems i found were related to ffmpeg and getid3,
with actual ffmpeg from CVS things got better. an update to
getid3 2.0 might improve the situation but probably another
way of handling formats not supported by getid3/ffmpeg has to be
introduced ogginfo / ffmpeg2theora would be candidates.

despite of these format problems i would consider the video extension
as quite stable. if some short term funding for hardware can be aquired
spektral.at and medienwahn.at people are planing to do a public
creative commons local short film archive based on the video node code
in january.

is there any test node for the payable extension yet?

from my point of view best would be to merge video extension and
payable code back to one trunk - with an on/off switch for video
file import/handling in the config (as some nodes want audio only
and every feature brings his bugs...)

> looking at these proceedings i would like to ask the question where
> to go from here. we do have a trunk and two branches. we're obviously
> going to merge our source into the video_test branch as this was our
> basis. then, how do we cope with the different versions? a move from
> the sysadmins to our video branch would be preferrable for the
> following reasons: - the video node can also be used as pure audio
> node (by switch in the config). - the video node will then have a WAI
> conform interface that can be styled via pure css - audio and video
> nodes can coexist in a network nevertheless it is important that all
> the nodes on the network update to the new version since only the new
> nodes (even if they provide audio only) can pass the video metadata
> on to other nodes. as soon as two video providing nodes are on the
> network, this is nescessary. we did quite some tests to proove that
> it is working.
> 
> what do you think is best? we obviously should discuss this in detail
> as the video extension is a major step in the network's history. it
> would probably be best if we set up a new test network on different
> locations to proove it extensively before we do the final move. we'll
> start with a clean node next week an will provide the source for you
> to test.

> finally, we'll need somebody with the access permission to the svn
