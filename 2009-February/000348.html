First of all please let me say that I think SOTF is a very well done 
piece of software. One of the rare examples for a successful project in 
that context! That's why people still recognize SOTF as a feasible 
solution, years after it was designed.

The existing documentation is valuable and fits the needs of people who 
want to install and administer a node. But people who see about possible 
implementation of SOTF into an existing system complain better (or maybe 
better structured) information about the certain functional parts of the 
code. That's the response I've got several times from coders. I'm truly 
not a PHP coder at all, I'm just reporting this feedback.

In radio's daily work there are 3 components operating the same data 
(and meta-data), have to fit together and collaborate as close as possible:

Broadcast plan generation + Playout system + Archive

In fact most radio stations already have one or two of this components 
running. So the ability of intergration into their running system is 
very important, and it-people think about possibly needed modifications 
of the code.

That's a good argument, maybe detailled documentation for coders is 
better kept outside the code.

> http://ec.europa.eu/education/lifelong-learning-programme/doc86_en.htm
> http://ec.europa.eu/education/grundtvig/doc868_en.htm
> 
> Scope of this project is the specification of requirements for 
> enhancements and refinements desired for SOTF. Grundtvig pays for 
> 'mobilities', which means workshops and meetings. This allows exhaustive 
> preparations for further development. The realisation (coding) has to be 
> done in a next step.
> 
> So the preparation of this proposal will not cost too much effort. 
> Deadline is 20th february, so time is quite short, but nevertheless i 
> think there is a good chance to keep it. At this moment we're waiting 
> for contribution of potential partners.
> 
> The partners of Grundtvig proposals have to be active on the field of 
> adult education. Does this apply to Sztaki? Hope so, we would highly 
> appreciate to have you enclosed.
> 
> Semantic web is a quite interesting matter, but not a central point from 
> the radio's view. Of course we need a more sophisticated and convenient 
> search, but this is also reachable on more simple ways. Implementation 
> of semantic web structures would bring a considerable touch of research 
> assignment into the project and requires a formidable expansion of it's 
> scope. Development closely based on the existing and well-proven code is 
> more on behalf of community radios. But this is just my opinion, if 
> semantic web is an utmost concern of a potential partner (Sztaki?) we 
> should discuss.
> 
> BTW - i'm currently in contact with some german community radios. In 
> fact they are interested in SOTF too, main argument for hesitation is 
> lack of comments within the PHP code. This is an issue to anyone who 
> thinks about integration of SOTF into an existing (or even planned) 
> system (means to almost everyone). IMHO it's quite important to improve 
> this, if we want to keep SOTF alive.
> 
> The community radio's main interest is focused on enhancements and 
> improvements based on the experiences of wokaday's life over the years. 
> Here is a list of certain issues:
> 
> Desirable Improvements:
> -----------------------
> 
> - Migration from now used PHP4 to recent version PHP5
> - Development of a more convenient and sophisticated search system
> - Enhancement of the existing user-administration
> - Implementation of hierarchical differentiated user roles
> - Grouping of accounts in order to build up editorial staffs
> - Special support for untreated o-tones, access for contributors only
> - Change for more convenient administration interfaces
> - Refinement of the existing RSS/PODCAST submission
> - Support for additional (particularly nordic) languages
> - Advancement of existing template and style-sheet management
> - Refinement of existing statistical analysis
> - Code cleanup and extension of code documentation
> - Implementation of LDAP interfaces
> - Enhancement of XBMF import/export functions
> - Development of interfaces to EUROPEANA database
> 
> To discuss:
> -----------
> 
> Emplacement of the facility to include or exclude certain series (or 
> even radio stations) from synchronisation. At this moment this is just 
> an idea to avoid interchange of conflicting and unwanted content. In 
> fact this is an issue, due to the enormous heterogeneity of community 
> radios. This has to be dicussed carefully, not only because of 
> increasing technical (and administrative) complexity, but also because 
> of the problematic closeness to censorship. Maybe this challenge is 
> better solved on the side of representation via RSS/PODACST.
> 
> Best regards
> Herbert
